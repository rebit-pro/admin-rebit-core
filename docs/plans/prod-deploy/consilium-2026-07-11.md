## Конфликты экспертов и их разрешение

### Общий вердикт по плану
Позиции: CI/CD-инженер — ready_with_fixes (каркас §6 честно списан с эталона, недостатки чинятся по ходу); Инфра, Безопасность, Релиз — not_ready
**Разрешение:** not_ready. Зона CI/CD-эксперта — самая узкая, и даже он запрещает исполнение §6 как есть (пайплайн не запустится и не соберётся). Блокеры остальных лежат вне его оптики и не чинятся «по ходу»: исполнение §13(4–5) как написано угрожает чужому продакшену, а осуществимость SSH-канала определяет архитектуру. Нюанс сохранён в порядке работ: repo-артефакты (аналог §13 шагов 1–3) можно делать параллельно ревизии плана — not_ready относится к плану как документу-инструкции, а не к запрету любых работ.

### Имя Swarm-стека
Позиции: CI/CD предлагает rebit-admin (по имени репозитория); Инфра и Релиз — admin (по конвенции коротких имён эталона: site, traefik)
**Разрешение:** admin: большинство (2:1), согласуется с уже принятым в плане именованием секретов admin_backend_env_<BUILD> (§8) и конвенцией эталона; на общем swarm уникально. Решение низкорисковое — критично лишь ОДНО имя везде: закрепить STACK_NAME ?= admin в Makefile и выправить §1/§3.3/§7 (сейчас три варианта: rebit-admin :37/:251, admin :253, rebit-admin_api :274).

### Модель токенов registry (TOKEN_GIT_HUB)
Позиции: CI/CD — оставить один PAT как в эталоне (write:packages в CI + read:packages на сервере), поправив формулировку; Безопасность — разделить: push через встроенный GITHUB_TOKEN (permissions: packages: write), на сервере — отдельный machine-PAT только read:packages через --password-stdin
**Разрешение:** Вариант Безопасности. Аргумент решающий: на shared-сервере PAT оседает plaintext в ~/.docker/config.json и виден соседнему проекту — write:packages там означает supply-chain-вектор (отравление образов админки). GITHUB_TOKEN эфемерный, штатно работает с docker/login-action, сложность не растёт. Механике пайплайна CI/CD-эксперта это не противоречит — меняется только источник токена в login-шаге и скоуп серверного PAT.

### Факт №3 исходных данных: «https://rebit-pro.ru → 200 — на сервере уже работает другой прод-сайт»
Позиции: Исходные факты консилиума приписывают 200 серверу 37.143.8.221; Инфра-эксперт утверждает, что 200 отдаёт GitHub Pages (DNS → 185.199.x.153), а сервер лишь держит LE-серт/роутер этого домена
**Разрешение:** Прав Инфра-эксперт — проверено синтезатором: getent hosts rebit-pro.ru → 185.199.108–111.153, curl -I https://rebit-pro.ru → server: GitHub.com. Материальный вывод НЕ меняется: живой сосед-прод на сервере подтверждён по app.rebit-pro.ru → 37.143.8.221 (P2P, 200) и api.rebit-pro.ru (401). Остаточная аномалия (свежий LE-серт rebit-pro.ru на сервере при DNS на GitHub Pages) — признак server-side drift и неучтённого стека; вопрос уходит в инвентаризацию.

### С чего начинать работы
Позиции: CI/CD ставит первым снятие противоречий в плане, затем артефакты, затем GitHub-side и SSH-канарейку; Инфра/Безопасность/Релиз требуют SSH-канарейку и инвентаризацию сервера нулевым шагом
**Разрешение:** Не или-или — параллелить. Правки плана не требуют сервера; канарейка и инвентаризация не требуют правок плана; оба трека — до серверных работ. Единственные жёсткие зависимости: финализация labels §3.3 и limits/reservations — после инвентаризации (фактические имена certResolver/middleware/сети и ёмкость хоста), а deploy-джоба пишется только после успешной канарейки (иначе архитектура меняется на self-hosted runner).

## Остаточные риски

- SSH с GitHub-hosted runners может оказаться закрыт (причина banner-timeout из среды аудита не установлена: geo/IP-фильтр, tarpit, fail2ban?) — тогда архитектура §6 меняется на self-hosted runner/relay; выяснится только на канарейке
- Server-side drift: фактический конфиг живого шлюза мог разойтись с репо /home/user/traefik (на сервере LE-серт rebit-pro.ru при DNS этого домена на GitHub Pages — признак неучтённого стека/истории); имена certResolver/middleware/entrypoints до инвентаризации не гарантированы — labels могут не заработать
- Ёмкость 37.143.8.221 под третий стек неизвестна (RAM/CPU/диск не обмерены): replicas:2 + reservations + Postgres + backup могут не влезть рядом с p2p и traefik
- Shared deploy-юзер: даже после минимизаций (read-only PAT, права 0700 на /srv) blast-radius между тремя проектами остаётся, если заказчик примет текущую модель tenancy
- PHP 8.5 в shivammathur/setup-php с pdo_pgsql не прогонялся из среды аудита; откат на 8.4 невозможен (composer требует ^8.5)
- Наличие/версия swarm-cronjob на живом swarm не подтверждены — cron-контур §7 может потребовать деплоя самого cronjob-сервиса, что не заложено
- Org-политики GitHub (разрешён ли classic PAT в rebit-pro, 2FA, права machine-аккаунта) из среды не видны — могут заблокировать выбранную токен-модель
- Лимит gha-кэша (10 GB/репо) на 4–5 образов не замерялся — возможно вытеснение кэша и медленные сборки
- 152-ФЗ-хвосты (Sentry-хостинг, оформление трансграничной передачи, если выберут SaaS) — организационные решения вне инженерного контроля, могут сдвинуть срок первого прод-релиза
