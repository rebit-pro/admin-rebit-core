# Сценарии работы с админкой ReBit Admin Core

> Документ описывает пользовательские и системные сценарии мини-админки.
> На нём основаны [`02-domain.md`](02-domain.md) (предметная область) и [`03-database.md`](03-database.md) (схема БД).
> Статус: проектный документ (design), сверен с фактическим кодом проекта на 2026-07-07.

## 0. Контекст и границы

**Что это.** Универсальная мини-админка: авторизация, управление собственной учётной записью, ролевой доступ, конструктор каталожных сущностей на базе упрощённых highload-блоков (по мотивам Bitrix, но проще), обработка доменных событий и фоновых задач по расписанию.

**Стек (факт из репозитория).** Slim 4, PHP 8.5, PHP-DI, PDO (без ORM), PostgreSQL 17 (локально — SQLite), самописный `Migrator`. Фронт — Vue 3 + Vuetify + Pinia (переносится из P2P, работает через `{data}`/`{error}`-контракт `JsonResponder`).

**Архитектурный уровень — L** (по [[rebit-backend]]): строгое разделение слоёв, DDD + гексагональная архитектура, каждый модуль имеет папки `Domain / Application / Infrastructure / Presentation`. Патоки команд/запросов — в стиле Елисеева (Command + Handler, Query + Fetcher), но на PDO-репозиториях-адаптерах.

**Что в MVP.** Аутентификация по **email + пароль**. Телефон как канал входа — заложен в модель, но не реализуется на этом этапе. OAuth2/OIDC, SSO, 2FA — вне MVP (точки расширения зафиксированы).

**Акторы.**

| Актор | Кто это | Доступ |
|---|---|---|
| **Гость (аноним)** | неавторизованный клиент | только публичные эндпоинты: вход, health |
| **Пользователь** (`role=user`) | рядовой сотрудник | свой профиль + операции согласно правам роли |
| **Администратор** (`role=admin`) | управляет контентом и пользователями | highload-блоки, элементы, пользователи (кроме владельца) |
| **Владелец** (`role=owner`) | суперпользователь | всё, включая управление ролями/правами и системными настройками |
| **Система** (cron / worker) | планировщик и обработчик событий | внутренние операции без HTTP-контекста |

> Роли `owner / admin / user` уже присутствуют в фикстуре `AuthFixture`. В MVP роль остаётся у пользователя одна; модель прав (RBAC) проектируется расширяемой (см. [`02-domain.md`](02-domain.md), контекст **Access**).

**Единый контракт ответа** (факт — `JsonResponder`):
- успех: `{ "data": ... }` (или `204 No Content`);
- ошибка: `{ "error": { "message": "...", "errors": { "поле": ["..."] } } }`.
- Коды: `200/201/204`, `400/401/403/404/409/422/429`, `500`.

## 1. Аутентификация и учётная запись

### СЦ-1.1. Вход по email и паролю
- **Актор:** Гость. **Эндпоинт:** `POST /api/v1/auth/login`. **Права:** нет.
- **Предусловия:** пользователь существует и активен.
- **Основной поток:**
  1. Гость отправляет `{ email, password }`.
  2. Система валидирует формат (email — корректный, пароль — непустой) → иначе `422`.
  3. Проверяется rate-limit по IP и по email (см. СЦ-6.1) → при превышении `429`.
  4. Ищется пользователь по email; сверяется пароль (`password_verify`, argon2id/bcrypt).
  5. При успехе генерируется **opaque-токен** (`base64url(random_bytes(48))`), в БД пишется только его `sha256`-хэш с TTL `+24h` (факт — `TokenFactory`, `auth_access_tokens`).
  6. Записывается событие `UserLoggedIn`, сбрасывается счётчик неудачных попыток.
  7. Ответ `200 { data: { token, expiresAt, user } }`.
- **Альтернативы / ошибки:**
  - неверные email/пароль → `401` `{error.message: "Неверный email или пароль"}`, инкремент счётчика неудач, событие `UserLoginFailed` (единый текст ошибки для неверного логина и пароля — не раскрываем, что именно неверно);
  - пользователь заблокирован/неактивен → `403`;
  - N неудач подряд (напр. 5 за 15 мин) → временная блокировка учётки (см. СЦ-6.2), `429`.
- **Безопасность:** CAPTCHA (GeeTest, уже интегрирована на фронте) подключается при подозрительной активности; ответ по времени выровнен (dummy-hash при отсутствии пользователя — защита от user-enumeration по таймингу).

### СЦ-1.2. Выход
- **Актор:** Пользователь. **Эндпоинт:** `POST /api/v1/auth/logout`. **Права:** аутентифицирован.
- **Поток:** по `Authorization: Bearer <token>` находится токен, удаляется его запись (по `token_hash`). Ответ `204`. Событие `UserLoggedOut`.
- **Опция:** `logout?all=true` — отзыв всех токенов пользователя (пригодится после смены пароля).

### СЦ-1.3. Текущий пользователь
- **Актор:** Пользователь. **Эндпоинт:** `GET /api/v1/auth/user`. **Права:** аутентифицирован.
- **Поток:** middleware аутентификации по Bearer-токену кладёт `Identity` в атрибут запроса; возвращается `200 { data: user }` (`id, email, login, name, role, phone, address`). Просроченный/неизвестный токен → `401`.

### СЦ-1.4. Смена пароля
- **Актор:** Пользователь (свой аккаунт). **Эндпоинт:** `POST /api/v1/account/change-password`. **Права:** аутентифицирован.
- **Основной поток:**
  1. Отправляется `{ currentPassword, newPassword, newPasswordConfirmation }`.
  2. Валидация: новый пароль удовлетворяет политике (мин. длина 8, не совпадает с текущим, подтверждение совпадает) → иначе `422`.
  3. Сверяется `currentPassword` → иначе `401`/`422` (`{errors.currentPassword}`).
  4. Пароль перехешируется и сохраняется; **все активные токены пользователя отзываются**, кроме текущего (или все — по политике); выдаётся новый токен.
  5. Событие `UserPasswordChanged`. Ответ `200 { data: { token, expiresAt } }`.
- **Ошибки:** неверный текущий пароль → `401`; слабый пароль → `422`.

### СЦ-1.5. Смена логина
- **Актор:** Пользователь. **Эндпоинт:** `POST /api/v1/account/change-login`. **Права:** аутентифицирован.
- **Предусловия:** логин отделён от email (факт — миграция `extend_auth_users_profile` добавила `login` с уникальным индексом).
- **Поток:**
  1. `{ login }` — валидация формата (латиница/цифры/`._-`, длина 3–32) → `422`.
  2. Проверка уникальности → занят → `409` `{errors.login}`.
  3. Обновление `login`, событие `UserLoginChanged`. Ответ `200 { data: user }`.

### СЦ-1.6. Смена email (с подтверждением)
- **Актор:** Пользователь. **Эндпоинт:** `POST /api/v1/account/change-email` + `POST /api/v1/account/change-email/confirm`. **Права:** аутентифицирован.
- **Поток:**
  1. `{ newEmail, currentPassword }` → валидация, проверка пароля, проверка что email свободен (`409`).
  2. Формируется код/токен подтверждения (запись с TTL), отправляется на **новый** email (в MVP — заглушка отправки, как текущий `REGISTRATION_CODE = 123456`).
  3. `confirm { code }` подтверждает и меняет email; событие `UserEmailChanged`.
- **Примечание:** email — это одновременно логин-идентификатор для входа. Поэтому смена email требует пароля и подтверждения. Смена «отображаемого логина» (СЦ-1.5) — легче, т.к. это не канал входа.

### СЦ-1.7. Регистрация пользователя (существует) и создание админом
- **Факт:** уже есть публичная регистрация по коду: `POST /api/v1/auth/register/request-code` → `POST /register/confirm` (код-заглушка `123456`).
- **Целевой сценарий админки:** саморегистрация может быть отключена; пользователей заводит **владелец/админ** (`POST /api/v1/users`, СЦ-2.1). Публичная регистрация остаётся как опциональный флаг настроек.

### СЦ-1.8. Восстановление пароля (точка расширения)
- `POST /api/v1/auth/forgot-password` → письмо с токеном → `POST /auth/reset-password`. Модель токенов подтверждения переиспользуется. В MVP — опционально; таблица/механика заложены в [`03-database.md`](03-database.md).

### (Заложено на будущее) Вход по телефону
- Модель `auth_users.phone` уже есть. Канал «вход по телефону + OTP» описан как расширение: тот же поток СЦ-1.1, но идентификатор — нормализованный телефон, фактор — одноразовый код по SMS. **В MVP не реализуется.**

## 2. Пользователи и доступ (RBAC)

### СЦ-2.1. Управление пользователями
- **Актор:** Владелец / Администратор. **Права:** `users.manage`.
- **Сценарии:** список (`GET /api/v1/users` с пагинацией/фильтром/поиском), создание (`POST`), просмотр (`GET /{id}`), редактирование профиля и роли (`PATCH /{id}`), блокировка/разблокировка (`POST /{id}/block`), удаление (`DELETE /{id}`).
- **Инварианты доступа (защита от IDOR):**
  - администратор **не может** менять/удалять владельца и повышать себя до владельца;
  - нельзя удалить/заблокировать последнего активного владельца;
  - пользователь без права `users.manage` видит и меняет **только свой** профиль.
- **События:** `UserCreated`, `UserRoleChanged`, `UserBlocked`, `UserDeleted`.

### СЦ-2.2. Роли и права
- **Актор:** Владелец. **Права:** `access.manage`.
- **Сценарии:** просмотр каталога прав (`GET /api/v1/permissions`), список ролей (`GET /roles`), назначение прав роли, назначение роли пользователю.
- **Модель:** роль → набор прав (permission-строки вида `catalog.manage`, `users.manage`, `elements.write`). Проверка прав — в middleware/Policy на каждое действие (не доверяем роли из запроса). В MVP набор ролей фиксирован (`owner/admin/user`), но структура таблиц позволяет добавлять роли без изменения кода.

## 3. Highload-блоки: конструктор каталожных сущностей

Цель — дать администратору инструмент «как инфоблоки в Битриксе, но проще»: создавать типы сущностей с настраиваемым набором полей и наполнять их элементами. На этом механизме строятся **каталог** и **офферы**.

**Термины** (детально — в [`02-domain.md`](02-domain.md)):
- **HL-блок (Block)** — тип сущности / справочник (напр. `catalog`, `offers`, `brands`). Имеет символьный код, имя, набор полей.
- **Поле (Field)** — свойство блока: символьный код, тип (`string, text, integer, decimal, boolean, datetime, enum, file, reference`), обязательность, настройки (длина, точность, значения enum, целевой блок для reference), участие в фильтре, сортировка.
- **Элемент (Element)** — запись блока. Системные поля (`id, active, sort, code, name, created_at, updated_at`) — реальные колонки; пользовательские значения — в `JSONB` (см. решение в [`03-database.md`](03-database.md)).

### СЦ-3.1. Создать HL-блок
- **Актор:** Администратор. **Эндпоинт:** `POST /api/v1/hl/blocks`. **Права:** `catalog.manage`.
- **Поток:** `{ code, name, description? }` → валидация (код уникален, `^[a-z][a-z0-9_]*$`) → создаётся блок → событие `BlockCreated`. Ошибки: код занят → `409`.

### СЦ-3.2. Управлять полями блока
- **Эндпоинты:** `POST /api/v1/hl/blocks/{block}/fields`, `PATCH .../fields/{field}`, `DELETE .../fields/{field}`.
- **Поток:** задаётся `{ code, type, required, multiple?, settings, showInFilter, sort }`. Система проверяет корректность настроек под тип (напр. `enum` требует список значений; `reference` — существующий целевой блок). Событие `FieldCreated/FieldUpdated/FieldDeleted`.
- **Инвариант:** удаление/смена типа поля, по которому уже есть данные, требует явного подтверждения (`?force=true`) — иначе `409`; значения приводятся/чистятся фоновой задачей (СЦ-5).

### СЦ-3.3. CRUD элементов
- **Эндпоинты:** `GET /api/v1/hl/blocks/{block}/elements` (список: пагинация, сортировка, фильтр по полям с `showInFilter`, полнотекстовый поиск по `name`), `POST`, `GET /{id}`, `PATCH /{id}`, `DELETE /{id}`.
- **Поток создания:** тело валидируется **динамически по определению полей блока** (обязательность, тип, диапазоны, enum-значения, существование reference-цели) → `422` с пофайловыми ошибками. Значения раскладываются: системные — в колонки, пользовательские — в `data JSONB`. Событие `ElementCreated`.
- **Инварианты:** значения reference-полей проверяются на существование целевого элемента (целостность связей — на уровне приложения, как в Bitrix HL-блоках); неактивные элементы (`active=false`) скрыты из публичной выдачи, но видны в админке.

### СЦ-3.4. Построить каталог и офферы (пример конфигурации)
Каталог и офферы — **не отдельный код**, а два настроенных блока:
1. Блок `catalog` (товары): поля `price(decimal)`, `brand(reference→brands)`, `sku(string)`, `description(text)`, `image(file)`, `in_stock(boolean)`.
2. Блок `offers` (торговые предложения): поле `product(reference→catalog)` + `price(decimal)`, `qty(integer)`, `min_amount(decimal)`, `payment_method(enum)`, `status(enum)`.
3. Связь «оффер → товар» — по `reference`-полю (по ID, как `Advertisement.UF_CURRENCY_PAIR_ID` в P2P).
- **Сценарий администратора:** создаёт блоки/поля (СЦ-3.1–3.2), наполняет товарами и офферами (СЦ-3.3), получает готовый каталог с торговыми предложениями. Для витрины фронт читает `GET /hl/blocks/catalog/elements?filter[in_stock]=true&sort=sort`.
- **Расширение:** если у каталога появится сложная доменная логика (расчёт цен, скидки), поверх движка добавляется тонкий модуль `Catalog` со своими use-case'ами — без дублирования хранилища.

## 4. События (Domain Events)

Механизм событий обязателен: доменные изменения порождают события, на которые подписаны обработчики (уведомления, аудит, инвалидация кэша, интеграции).

### Модель (см. [`02-domain.md`](02-domain.md) §8 и [`03-database.md`](03-database.md))
**Итог консилиума: в MVP — только синхронная in-process шина событий**, разведённая по фазе относительно коммита:
- **`in-transaction`-подписчики** (аудит) — в той же транзакции, что и операция: нет действия без записи в журнал (падение аудита откатывает операцию).
- **`after-commit`-подписчики** (инвалидация кэша, интеграции) — строго после коммита; их ошибка не откатывает операцию.
- Транзакционную границу задаёт UnitOfWork/декоратор в `Shared`.
- **Transactional outbox (`event_outbox`) и асинхронная доставка отложены до MVP-2** — в MVP нет ни одного async-потребителя. Порт `EventPublisher` позволяет добавить outbox и брокер (RabbitMQ/Redis) позже без изменения доменного кода; тогда же вводится consumer-side идемпотентность по `(event_id, handler)`.

### СЦ-4.1. Публикация события
- **Актор:** Система (внутри Handler). **Поток:** Handler после изменения агрегата вызывает `EventDispatcher::dispatch(event)`; синхронные подписчики отрабатывают сразу, событие сериализуется в `event_outbox`. Пример: `ElementCreated` → синхронно пишет `audit_log`, асинхронно шлёт webhook-интеграцию.

### СЦ-4.2. Подписка / обработка
- Реестр подписок `event_subscriptions` (тип события → обработчик, вкл/выкл). Ошибка обработчика не откатывает основную транзакцию; для async — ретраи с backoff и лимитом, после лимита запись помечается `failed` (dead-letter).

**Каталог доменных событий (базовый):** `UserLoggedIn`, `UserLoginFailed`, `UserLoggedOut`, `UserPasswordChanged`, `UserLoginChanged`, `UserEmailChanged`, `UserCreated`, `UserRoleChanged`, `UserBlocked`, `UserDeleted`, `BlockCreated`, `FieldCreated/Updated/Deleted`, `ElementCreated/Updated/Deleted`.

## 5. Планировщик и фоновые задачи (Cron)

Долгие/периодические операции выполняются **вне HTTP-цикла** консольными командами, запускаемыми планировщиком (системный `cron` / `supercronic` в контейнере — как в P2P). CLI-точка входа — `bin/app.php` (факт: сейчас `match` по `$argv`; предлагается перевести на `symfony/console` для команд с аргументами/расписанием).

### Механика (см. [`03-database.md`](03-database.md))
- Каждая задача — консольная команда с **защитой от параллельного запуска** (лок в таблице `cron_locks` или `symfony/lock`), логированием старта/финиша и результата в `job_runs`.
- Реестр расписаний — `cron_jobs` (код команды, cron-выражение, вкл/выкл, последний запуск), опционально исполняется единым диспетчером `schedule:run` (запускается системным cron ежеминутно).

### Базовые задачи
| Команда | Расписание | Назначение | СЦ |
|---|---|---|---|
| `events:dispatch-outbox` | каждую минуту | публикация событий из `event_outbox` подписчикам | СЦ-5.2 |
| `auth:purge-expired-tokens` | ежечасно | удаление протухших `auth_access_tokens` и кодов подтверждения | СЦ-5.3 |
| `hl:reindex` | по требованию/ночью | пересчёт/чистка значений после изменения полей (СЦ-3.2) | — |
| `audit:rotate` | ежедневно | архивация/чистка старого аудита | — |
| `schedule:run` | ежеминутно (системный cron) | диспетчер, запускающий задачи по `cron_jobs` | — |

### СЦ-5.1. Запуск задачи по расписанию
- **Актор:** Система. **Поток:** cron вызывает `php bin/app.php schedule:run` → диспетчер выбирает задачи, чьё время наступило и которые не заблокированы → берёт лок → выполняет → снимает лок, пишет `job_runs`. Двойной запуск невозможен (лок).

### СЦ-5.2. Обработка outbox
- Выбирает пачку неопубликованных событий (`status=pending`, `FOR UPDATE SKIP LOCKED`), вызывает подписчиков, помечает `published`/`failed` (с ретраями). Идемпотентно по `event_id`.

### СЦ-5.3. Очистка токенов
- Удаляет записи `auth_access_tokens.expires_at < now()` и просроченные коды подтверждения. Снижает поверхность атаки и размер таблиц.

## 6. Безопасность (защита от внешних угроз, OWASP-минимум)

Требование обязательное. Ниже — сценарии защиты, сопоставленные с OWASP Top-10, реализуемые в основном как middleware в HTTP-ядре и правила в доменных слоях.

### СЦ-6.1. Rate limiting
- На публичные и auth-эндпоинты (`/auth/login`, `/auth/register/*`, `/account/*`) — ограничение по IP и по идентификатору (email). Превышение → `429` с `Retry-After`. Хранилище счётчиков — таблица/кэш (в MVP — БД; при Redis — туда). *(В референсе Елисеева rate-limit в приложении отсутствует — мы его добавляем.)*

### СЦ-6.2. Защита от перебора паролей (brute force)
- Счётчик неудачных попыток на пользователя/IP; после порога — временная блокировка учётки и требование CAPTCHA. Событие `UserLoginFailed` фиксируется в аудите; при аномалии — `SecurityAlert`.

### СЦ-6.3. Аутентификация и авторизация
- Все непубличные маршруты — за middleware аутентификации (Bearer opaque-токен, хэш в БД, TTL, отзыв). Проверка прав (RBAC/Policy) на **каждое** действие. Проверка владельца ресурса — защита от **IDOR** (напр. пользователь не читает чужой профиль по подставленному `id`).

### СЦ-6.4. Валидация и защита от инъекций
- **Вся** входная валидация — на сервере (даже если фронт валидирует). Строгая денормализация DTO: неизвестные поля отклоняются (защита от **mass-assignment**). Только prepared statements / параметризованные запросы (факт — PDO с плейсхолдерами), **никакой конкатенации SQL** → защита от **SQL-injection**. Для `JSONB`-значений — типовая валидация по определению поля.

### СЦ-6.5. Заголовки и транспорт
- Security-заголовки: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy`, `Content-Security-Policy` (для админ-фронта), HSTS на проде. **CORS по белому списку** источников (не `*` в проде). *(В референсе CORS сделан на Nginx с `*` — на проде сужаем.)*

### СЦ-6.6. Данные и секреты
- Пароли — `password_hash` (argon2id/bcrypt), в БД только хэши; токены — только `sha256`-хэш (факт). Секреты — только в `.env`, не в коде/гите; `APP_DEBUG=false` на проде; ошибки логируются, наружу — обобщённые сообщения (без стектрейсов).

### СЦ-6.7. Загрузка файлов (для поля типа `file`)
- Валидация типа/размера, хранение вне webroot, генерация имён, запрет исполняемых типов.

### СЦ-6.8. Аудит и наблюдаемость
- Значимые действия (вход, смена пароля/логина/email, изменения пользователей/прав, CRUD блоков/элементов) пишутся в `audit_log` (кто, что, когда, до/после, IP). Health-check `GET /health` (факт) для мониторинга. Логирование через Monolog (пакет подключён — включить).

### Сопоставление с OWASP Top-10
| Риск | Мера (сценарий) |
|---|---|
| A01 Broken Access Control | RBAC + Policy на каждое действие, проверка владельца (IDOR) — СЦ-6.3 |
| A02 Cryptographic Failures | argon2id пароли, хэш токенов, HTTPS/HSTS — СЦ-6.5/6.6 |
| A03 Injection | prepared statements, серверная валидация — СЦ-6.4 |
| A04 Insecure Design | доменные инварианты, transactional outbox, лимиты — СЦ-4/6 |
| A05 Security Misconfiguration | `APP_DEBUG=false`, security-заголовки, CORS-whitelist — СЦ-6.5 |
| A07 Auth Failures | rate-limit, анти-brute-force, TTL/отзыв токенов — СЦ-6.1/6.2 |
| A08 Data Integrity | идемпотентность outbox, подтверждение смены email — СЦ-4/1.6 |
| A09 Logging Failures | аудит + Monolog + SecurityAlert — СЦ-6.8 |

### СЦ-6.9. Дополнения по итогам консилиума (обязательны до прода)
- **Redaction в аудите и событиях:** запрещено писать в `audit_log`/payload событий `password_hash`, токены, `code_hash`, сырые новые пароли — только whitelist безопасных полей. `auth_confirmations.payload` с «хэшем пароля до подтверждения» — под тем же ограничением.
- **Коды подтверждения:** криптослучайный код (не стаб `123456` — он недопустим на проде, в DoD), **лимит попыток на `confirm` (напр. 5)** с инвалидацией, короткий TTL. Смена email без этого = вектор захвата аккаунта.
- **Anti-enumeration на `409`:** эндпоинты «занятости» (смена логина/email, request-code) не должны раскрывать существование учётки синхронно — нейтральный ответ + rate-limit/капча на `/account/*`.
- **152-ФЗ / ПДн (email, телефон, адрес, ФИО, IP):** согласие субъекта (таблица `user_consents` + версия политики + timestamp/IP), политика и **сроки хранения/удаления** ПДн (право на удаление/обезличивание), обезличивание ПДн в аудите по истечении срока.
- **GeeTest → заменить на РФ-капчу (решение).** Рекомендация: **Yandex SmartCaptcha** (РФ-хостинг, 152-ФЗ-совместимо, бесплатна, есть невидимый режим) вместо GeeTest (КНР → трансграничная передача ПДн). Домены капчи — в CSP allowlist; внести обработчика в перечень ПДн. GeeTest оставлять только при явной причине с оформленным основанием трансграничной передачи.
- **Шифрование at-rest:** SSE на S3-бэкапах (шифрование дампов), шифрование тома БД; ротация ключей.
- **Sentry:** `send_default_pii=false` + `before_send`-санитизация — Sentry не должен получать ПДн из трейсов.
- **CSP:** конкретная политика без `unsafe-inline`, allowlist GeeTest; при хранении токена в `localStorage` — компенсировать строгим CSP (или httpOnly-cookie).
- **security_alerts:** минимальный канал алертинга (email/Sentry) на `severity=critical`. В MVP допустимо вести как `audit_log.action='security.*'`; отдельная таблица — при вводе SIEM.

## 8. Принятые решения

Открытые вопросы закрыты (рекомендации приняты как решения проекта; детали — в [`02-domain.md`](02-domain.md) и [`03-database.md`](03-database.md)).

1. **Хранение значений HL-элементов — гибрид «системные колонки + `JSONB`»** (подход C). Не EAV и не таблица-на-блок: без runtime-DDL, гибкие поля, GIN/expression-индексы PG17. Материализация отдельного блока в реальные колонки — как точечная оптимизация позже.
2. **ORM — остаёмся на PDO** с репозиториями-адаптерами за портами домена (консистентно с текущим модулем `Auth`). Doctrine — как возможная будущая замена без правки домена (порты неизменны).
3. **Консоль — вводим `symfony/console`** вместо `match($argv)`. Нужно для аргументов, кодов возврата, `schedule:run` и cron-команд (СЦ-5).
4. **Саморегистрация — выключена по умолчанию** (флаг системных настроек). Пользователей заводит владелец/админ (СЦ-2.1); публичная регистрация по коду остаётся опцией.
5. **Роли — одна роль на пользователя в MVP** (поле `auth_users.role`). Схема RBAC готова к many-to-many (`access_user_roles`) — включается без изменения API.
6. **БД — единая PostgreSQL 17** для dev и прод; SQLite — только для быстрых юнит-тестов. Двойная ветка миграций упраздняется после ввода `pdo_pgsql` в образ ([`04-devops.md`](04-devops.md) §13).
7. **Инфраструктура без Redis на старте:** счётчики rate-limit, локи и outbox — в Postgres; Redis вводится при росте нагрузки.
