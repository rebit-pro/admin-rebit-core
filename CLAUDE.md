# ReBit Admin Core — Coding Context

Отвечать на русском. Senior PHP: SOLID, KISS, DRY, без оверкодинга.
На вопрос сначала отвечай, потом меняй код (если попросили). Перед нетривиальной правкой — план и подтверждение.
«Утверждаю / ок / делаем так» — согласование, не команда на коммит (см. Git). Баги/опечатки не по теме — тоже сообщай.

## Stack

Монорепо: `api/` (backend) + `frontend/` (SPA) + `docs/` (база знаний проекта).

- **API:** PHP 8.5, Slim 4, PHP-DI, PostgreSQL 17, Monolog, Symfony Console/Uid. Namespace `App\` → `api/src/`, `App\Test\` → `api/tests/`
- **Frontend:** Vue 3 + TypeScript, Vite, Vuetify (Berry), Pinia, Vue Router, vee-validate, vue-i18n
- **Окружение:** Docker Compose только для локальной разработки

## PHP Style

- `declare(strict_types=1)` везде, строгая типизация (аргументы, return, свойства)
- Yoda style: `null === $value`, `'' !== $string`
- Классы по умолчанию `final`. Добавляй `readonly`, если у класса есть конструктор (и, соответственно, состояние в свойствах). Без конструктора `readonly` не нужен
- `match` вместо `switch`
- Enum case: `case FIRST_ELEMENT = 'firstElement'`
- Явные проверки вместо `empty()`: `[] === $array`, `null === $value`. Исключение: `!$bool`, конкатенация
- Cast без пробела: `(int)$value`; PSR-12 скобки метода на новой строке
- Замыкания/стрелки: типы аргументов, возврата, `static` если применимо
- phpDoc массивов в стиле phpStan с переносом:

```php
/** @var array<int, array{
 *     id: int,
 *     name: string,
 * }> */
```

## Архитектура

Модули в `api/src/`: `Auth`, `Access`, `Audit`, `Http`, `Shared`. Внутри модуля слои: `Domain`, `Application`, `Infrastructure`, `Presentation`.

- Зависимости — через конструктор (property promotion); DTO/VO `final readonly` + named args
- Инфра-исключения → предметные исключения Application/Domain
- Hot path: без лишних циклов, `array_merge`, spread, промежуточных массивов

**Выбор слоя:** БД / HTTP-клиенты / внешние сервисы / security-механика → `Infrastructure`; HTTP-actions / API-ответ → `Presentation/Http`; предметное правило → `Domain`; сценарий через порты → `Application` (`Command` / `Query` / `Port`). Вся бизнес-логика — в Application-хендлерах, не в actions.

**Антипаттерны:** SQL в Application-хендлере, бизнес-логика в HTTP-action, форматирование ответа в Application, прямой доступ к Infrastructure из Presentation мимо портов.

**DI:** `api/config/container.php` подключает `api/config/di/*`. Сервисы — без изменяемого состояния (данные — через аргументы методов). Маршруты — `api/config/routes.php`.

## Workflow

Изменение не считается завершённым, пока не выполнены: линт изменённого кода + релевантный прогон тестов в Docker.

```bash
make up / down / restart / build / logs   # жизненный цикл
make api-migrate                          # миграции
make api-fixtures                         # фикстуры
make api-test                             # PHPUnit в контейнере
make api-lint                             # php -l по src/config/tests/bin/public
make api-fixer                            # php-cs-fixer fix (конфиг api/.php-cs-fixer.dist.php)
make api-cli                              # shell в контейнере app
make frontend-check                       # vue-tsc typecheck
make frontend-build                       # production-сборка
```

Frontend дополнительно: `npm run check` (lint + typecheck + typecheck:e2e), `npm run test:e2e:run` (Cucumber + Playwright).

## Git

### Формат коммита

Описание на русском. Никогда не добавляй себя в соавторы в описании коммита.

### Формат названия миграции

`YYYYMMDDHHMMSS_short_description.sql` в `api/migrations/` — дата и время создания → короткое описание на английском через подчёркивания.

### Когда коммитить

- **Только по явной просьбе пользователя.** Явная просьба — фраза вида «коммить», «закоммить», «сделай коммит», «коммитим», «commit», «смержь», «смерджи», «push».
- **Не коммит:** «утверждаю», «ок», «делаем так», «да», «согласен» — это согласование правки, не команда на коммит/мердж. Завершил правку — остановись и жди отдельной просьбы.
- **Исключение:** если пользователь явно сказал «работаем без подтверждения по этому плану» (или эквивалент), по всему этому плану можно коммитить промежуточные этапы автономно. Область действия — только тот план/задача, про который это сказано.
    - **Граница исключения по времени:** автономия действует только до конца последнего шага исходного плана. Все шаги реализованы и зафиксированы — исключение исчерпано, дальше снова «коммит только по явной просьбе».
    - **Подсказки/замечания/правки от пользователя после завершения плана** — это отдельные правки, не продолжение плана. Делать правку → остановиться → ждать новой явной просьбы на коммит.
    - **Если непонятно**, входит ли новая правка в исходный план — спросить, прежде чем коммитить.

### Удалённый git

Никогда не лезь в remote/GitHub без явного разрешения: `git push` / `--force` / `--delete`, `gh pr ... / issue ...`, переименование/удаление веток на origin. Локальные операции (`commit`, `merge`, `branch -m` локально) — по правилу «явная просьба» выше.

`CLAUDE.md` правим только по явной просьбе.
